# 🎤 สคริปต์นำเสนอ 5 นาที — Git และระบบควบคุมเวอร์ชัน

> **หัวข้อ**: การใช้ Git และระบบควบคุมเวอร์ชัน (Version Control Systems)
> **เว็บ Demo**: Chicken Shop — เว็บร้านไก่ทอด deploy บน Ubuntu + Nginx
> **Flow**: branch → edit → commit → push → PR → merge → deploy → เว็บเปลี่ยน

## SECTION 1 — เปิดเรื่อง (40 วินาที)

### 🎯 พูดว่า:

> "สวัสดีครับ วันนี้ผมจะมาสาธิตเรื่อง **Git และระบบควบคุมเวอร์ชัน**
>
> ลองนึกภาพว่าเราทำงานในทีม 5 คน แก้ไขไฟล์เดียวกัน —
> ถ้าไม่มีระบบควบคุม จะเกิดปัญหา **ไฟล์ทับกัน แก้แล้วพัง ย้อนกลับไม่ได้**
>
> Git แก้ปัญหานี้ โดยทำให้เรา **แยกงาน, บันทึกทุกเวอร์ชัน, review ก่อนรวม, และย้อนกลับได้เสมอ**
>
> ผมจะสาธิตสดด้วยเว็บนี้ครับ — **Chicken Shop** ร้านไก่ทอด
> deploy อยู่บน Ubuntu + Nginx ตอนนี้อยูลปามซ์เซิร์ฟเวอร์จริง"

### 🖱 ทำ:
1. **เปิดเบราว์เซอร์** → โชว์เว็บ Chicken Shop ที่ deploy อยู่
2. ชี้ให้เห็น section โปรโมชั่นปัจจุบัน (อันนี้เราจะเปลี่ยนมัน!)

## SECTION 2 — สร้าง Branch (50 วินาที)

### 🎯 พูดว่า:

> "ในองค์กรจริง เราจะ **ไม่แก้โค้ดบน main โดยตรง**
> เพราะ main คือเวอร์ชันที่ deploy อยู่ — ถ้าแก้แล้วพัง เว็บล่มทันที
>
> เราจึงต้อง **สร้าง branch** แยกออกมา เหมือนแยกสำเนาออกมาแก้
> จะแก้อะไรก็ได้ ไม่กระทบเว็บที่ออนไลน์อยู่"

### 🖱 ทำ: (เปิด Terminal)

```bash
# โชว์ว่าตอนนี้อยู่ branch main
git branch

# สร้าง branch ใหม่สำหรับแก้โปรโมชั่น
git checkout -b feature/update-promotion
```

### 🎯 พูดต่อ:

> "ตอนนี้ผม **แยก branch ออกมา** เรียบร้อย
> ทุกอย่างที่ผมแก้จะอยู่ใน branch นี้ ไม่ไปกระทบ main"

---

## SECTION 3 — แก้ไข + Commit + Push (60 วินาที)

### 🎯 พูดว่า:

> "ทีนี้ผมจะแก้ไขโปรโมชั่นบนเว็บครับ"

### 🖱 ทำ:

1. **เปิด `index.html`** ใน editor
2. **เลื่อนหาคำว่า "EASY EDIT POINT"** ในส่วน Promotion
3. **แก้โปรโมชั่น** เช่น:
   - เปลี่ยน `สั่ง 3 ชิ้น ลด 20%` → `ซื้อ 2 แถม 1!`
   - เปลี่ยน `หมดเขต 31 มี.ค.` → `เฉพาะสัปดาห์นี้`
4. **Save ไฟล์**

### 🎯 พูดว่า:

> "แก้เสร็จแล้ว ต่อไปผมจะ **commit** คือบันทึกการเปลี่ยนแปลงนี้เข้า Git
> และ **push** ส่งขึ้น GitHub"

### 🖱 ทำ: (Terminal)

```bash
# ดูว่าแก้ไขอะไรไปบ้าง
git diff

# เตรียมไฟล์ + บันทึกเวอร์ชัน
git add .
git commit -m "อัปเดตโปรโมชั่น: ซื้อ 2 แถม 1"

# ส่งขึ้น GitHub
git push origin feature/update-promotion
```

### 🎯 พูดว่า:

> "ตอนนี้โค้ดอยู่บน GitHub แล้ว แต่ **ยังไม่ได้รวมเข้า main**
> เว็บยังไม่เปลี่ยน เพราะเรายังไม่ได้อนุมัติ"

---

## SECTION 4 — Pull Request + Merge (60 วินาที)

### 🎯 พูดว่า:

> "ขั้นตอนต่อไปคือ **Pull Request** หรือ PR
> คือการขอรวมโค้ดเข้า main โดยให้คนในทีม **review** ก่อน
> เหมือนกับพนักงานเขียนรายงานเสร็จ ต้อง **ส่งหัวหน้าอนุมัติ** ก่อนจะประกาศ"

### 🖱 ทำ: (เปิด GitHub บน browser)

1. เปิด repo → จะเห็นแถบ **"Compare & pull request"** → **กดเลย**
2. ใส่ชื่อ PR: `อัปเดตโปรโมชั่น: ซื้อ 2 แถม 1`
3. กด **"Create pull request"**

### 🎯 พูดว่า:

> "ตอนนี้ทีมสามารถ review โค้ดได้ก่อน merge
> ในองค์กรจริง ถ้าโค้ดมีปัญหา หัวหน้าจะ reject ได้
> แต่วันนี้ผมจะ **approve แล้ว merge** เลย"

### 🖱 ทำ:

4. กด **"Merge pull request"** → **"Confirm merge"**

### 🎯 พูดว่า:

> "merge สำเร็จแล้ว! ตอนนี้โค้ดใหม่ **รวมเข้า main** แล้ว
> ทีนี้มาดูว่า... เว็บจะเปลี่ยนเองไหม?"

---

## SECTION 5 — Deploy ขึ้นเซิร์ฟเวอร์ (50 วินาที) ⭐ จุดว้าว

### 🎯 พูดว่า:

> "พอ merge เข้า main แล้ว ต่อไปก็นำโค้ดล่าสุดไปอัปเดตบนเซิร์ฟเวอร์
> ในองค์กรที่เซิร์ฟเวอร์มี public IP จะใช้ CI/CD ทำอัตโนมัติ
> แต่เซิร์ฟเวอร์ของเราเป็น IP ภายใน ผมจะ SSH เข้าไป pull สดให้ดูเลย"

### 🖱 ทำ: (Terminal)

```bash
# SSH เข้าเซิร์ฟเวอร์ Ubuntu (password: 1)
ssh analyze@192.168.0.111

# เข้าไปที่โฟลเดอร์เว็บ
cd /var/www/html

# git fetch = ดึงข้อมูลล่าสุดจาก GitHub มา (ยังไม่แก้ไฟล์)
git fetch origin

# git reset --hard = บังคับให้ไฟล์บนเซิร์ฟเวอร์ตรงกับ GitHub เป๊ะๆ
git reset --hard origin/main
```

### 🖱 ทำ:

1. **กลับไป browser ที่เปิดเว็บไว้ → กด Refresh**

### 🎯 พูดว่า:

> **(ชี้จอ)** "เว็บเปลี่ยนแล้วครับ! โปรโมชั่นเปลี่ยนเป็น 'ซื้อ 2 แถม 1' ตามที่แก้ไป
>
> ทั้งหมดนี้: แก้โค้ด → push → merge → pull บนเซิร์ฟเวอร์ → เว็บอัปเดต
> ในองค์กรจริงขั้นตอน deploy นี้จะทำอัตโนมัติด้วย CI/CD ครับ"

## SECTION 6 — สรุปปิดท้าย (40 วินาที)

### 🎯 พูดว่า:

> "สรุปครับ สิ่งที่ผมสาธิตไป มี **6 ขั้นตอนหลัก**"

**(ถ้ามี slide ก็โชว์ หรือพูดเลยก็ได้)**

> "1. **Branch** — แยกงานออกมา ไม่กระทบ production
> 2. **Edit** — แก้ไขไฟล์
> 3. **Commit** — บันทึกเวอร์ชัน สามารถย้อนดูได้ตลอด
> 4. **Push** — ส่งขึ้น GitHub ทำงานร่วมกันในทีม
> 5. **Pull Request** — review โค้ดก่อน merge ป้องกันบั๊ก
> 6. **Merge + Auto Deploy** — รวมโค้ดแล้ว deploy อัตโนมัติ
>
> สิ่งเหล่านี้คือพื้นฐานที่ **ทุกบริษัท tech ใช้อยู่ทุกวัน**
> ไม่ว่าจะเป็น Google, Facebook, LINE หรือบริษัทในไทย
> Git ช่วยให้ทีมทำงานร่วมกัน ไม่ทับกัน ย้อนกลับได้ และ deploy ปลอดภัย
>
> ขอบคุณครับ"

## 🛡 เตรียมตัวตอบ Q&A

อาจารย์อาจถามคำถามเหล่านี้ เตรียมคำตอบไว้:

### ❓ "ทำไมต้องใช้ branch? แก้ใน main เลยไม่ได้เหรอ?"
> "ได้ครับ แต่ถ้าแก้แล้วพัง เว็บจะล่มทันที เพราะ main คือสิ่งที่ deploy อยู่
> การแยก branch ทำให้เราทดสอบก่อนได้ ถ้ามีปัญหาก็ไม่กระทบ production"

### ❓ "Pull Request มีประโยชน์อะไร?"
> "ให้คนอื่นในทีม review โค้ดก่อน merge เหมือนระบบอนุมัติ
> ถ้าโค้ดมีบั๊ก หัวหน้าสามารถ comment หรือ reject ได้ ไม่ปล่อยของพังขึ้น production"

### ❓ "ถ้า deploy แล้วพัง ทำยังไง?"
> "ใช้ git revert ย้อน commit ล่าสุดกลับ แล้ว push อีกรอบ
> GitHub Actions จะ deploy เวอร์ชันที่ถูกต้องกลับมาอัตโนมัติ
> เพราะ Git เก็บทุกเวอร์ชันไว้หมด เราสามารถย้อนกลับไปเวอร์ชันไหนก็ได้"

### ❓ "GitHub Actions คืออะไร?"
> "คือระบบ automation ของ GitHub ที่เราตั้งกฎไว้ว่า
> ถ้ามี push เข้า main ให้ SSH ไปเซิร์ฟเวอร์แล้ว pull โค้ดใหม่ให้อัตโนมัติ
> เรียกว่า CI/CD ครับ ในองค์กรจริงจะมีทั้ง test + build + deploy อัตโนมัติ"

### ❓ "Git กับ GitHub ต่างกันยังไง?"
> "Git คือตัวโปรแกรม version control ที่รันบนเครื่องเรา
> GitHub คือ platform ออนไลน์ที่เก็บ Git repo ไว้บน cloud
> เหมือน Git เป็นกล้อง ส่วน GitHub เป็น Google Photos ที่เก็บรูปไว้ให้"
